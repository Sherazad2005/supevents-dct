# Vue des processus

## Introduction

Cette section présente les principaux processus métiers de SupEvents à travers plusieurs diagrammes de séquence. Ces diagrammes décrivent les échanges entre les différents composants du système lors des parcours critiques de l’application.

Ils permettent d’analyser les flux synchrones, les traitements asynchrones ainsi que la gestion des erreurs liées au paiement.

---

#### Diagramme de composants — API Backend

Ce diagramme représente les principaux modules internes du backend SupEvents ainsi que leurs dépendances fonctionnelles et techniques.

```mermaid
flowchart TD

auth["AuthModule"]
event["EventModule"]
ticket["TicketModule"]
payment["PaymentModule"]
notif["NotificationModule"]
user["UserModule"]

stripe["Stripe"]
rabbit["RabbitMQ"]
postgres["PostgreSQL"]
redis["Redis"]

ticket --> event
ticket --> payment

payment --> stripe
payment --> rabbit

notif --> rabbit

auth --> redis
auth --> postgres

event --> postgres
ticket --> postgres
user --> postgres
```

### Lecture du diagramme

Le diagramme met en évidence le découplage entre les traitements métier et les services externes. Le module Ticket dépend des modules Event et Payment pour vérifier les disponibilités et traiter les paiements.

RabbitMQ permet d’isoler les notifications asynchrones du reste de l’application afin de limiter le couplage entre composants.

---

#### Diagramme de séquence — Inscription à un événement

Ce diagramme décrit le parcours nominal d’un étudiant souhaitant s’inscrire à un événement via SupEvents.

```mermaid
sequenceDiagram

actor Etudiant

participant Frontend
participant Gateway
participant AuthModule
participant EventModule
participant TicketModule
participant PaymentModule
participant PostgreSQL
participant Stripe
participant RabbitMQ
participant NotificationModule

Etudiant->>Frontend: Demande inscription
Frontend->>Gateway: HTTPS Request
Gateway->>AuthModule: Vérification token
AuthModule-->>Gateway: Token valide

Gateway->>EventModule: Vérifier places disponibles
EventModule->>PostgreSQL: SELECT places
PostgreSQL-->>EventModule: Places disponibles

Gateway->>TicketModule: Créer réservation pending
TicketModule->>PostgreSQL: INSERT réservation
PostgreSQL-->>TicketModule: Réservation créée

Gateway->>PaymentModule: Initier paiement
PaymentModule->>Stripe: HTTPS Paiement

alt Paiement accepté
Stripe-->>PaymentModule: Confirmation paiement

PaymentModule->>TicketModule: Confirmer ticket
TicketModule->>PostgreSQL: UPDATE confirmed

TicketModule->>RabbitMQ: Event TicketConfirmed

RabbitMQ-->>NotificationModule: Consommer événement
NotificationModule-->>Etudiant: Email + QR Code

else Paiement refusé
Stripe-->>PaymentModule: Erreur paiement
PaymentModule->>TicketModule: Annuler réservation
end
```

### Lecture du diagramme

Le processus montre une séparation claire entre les traitements synchrones liés à la réservation et les traitements asynchrones liés aux notifications.

L’utilisation de RabbitMQ permet de ne pas bloquer le parcours utilisateur pendant l’envoi des emails transactionnels.

#### Diagramme de séquence — Échec de paiement

Ce diagramme représente les différents scénarios d’échec lors du paiement d’un événement.

```mermaid
sequenceDiagram

actor Etudiant

participant PaymentModule
participant Stripe
participant TicketModule
participant PostgreSQL
participant RabbitMQ
participant NotificationModule

PaymentModule->>Stripe: Demande paiement

alt Timeout Stripe
Stripe-->>PaymentModule: Aucune réponse

PaymentModule->>TicketModule: Annuler réservation
TicketModule->>PostgreSQL: UPDATE cancelled

PaymentModule->>RabbitMQ: Event PaymentTimeout
RabbitMQ-->>NotificationModule: Notification timeout

else Carte refusée
Stripe-->>PaymentModule: Paiement refusé

PaymentModule->>TicketModule: Annuler réservation
TicketModule->>PostgreSQL: UPDATE cancelled

PaymentModule->>RabbitMQ: Event PaymentFailed
RabbitMQ-->>NotificationModule: Notification échec paiement

end
```

### Lecture du diagramme

Les deux scénarios d’échec entraînent la libération de la réservation afin d’éviter le blocage inutile des places disponibles.

Le découplage via RabbitMQ permet également de centraliser la gestion des notifications et du logging des incidents de paiement.
