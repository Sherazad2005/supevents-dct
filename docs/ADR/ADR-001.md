# ADR-001 — La concurrence sur la jauge de places est gérée par verrou pessimiste PostgreSQL

**Statut** : Accepté  
**Date** : 2025-05-13  
**Auteurs** : Sherazad

---

## Contexte

SupEvents permet à un nombre illimité d'utilisateurs de tenter de s'inscrire simultanément à un événement dont la capacité est strictement limitée (`seats_available`). Les ouvertures de billetterie populaires génèrent des pics de requêtes concurrentes en quelques secondes.

Le problème central est la garantie d'invariant : à aucun moment `seats_available` ne doit devenir négatif, et deux utilisateurs ne doivent pas obtenir le même "dernier ticket". Cette garantie doit être assurée sans fenêtre de race condition, même sous charge.

Trois forces s'exercent en sens contraire : la correction (aucun survente), la latence (l'utilisateur attend la confirmation), et la complexité opérationnelle (l'équipe est petite, deux développeurs).

---

## Décision

Nous utilisons un verrou pessimiste PostgreSQL via `SELECT ... FOR UPDATE` sur la ligne `Event` concernée, à l'intérieur d'une transaction qui englobe le décrément de `seats_available` et la création du `Ticket`. Toute la logique d'allocation s'exécute en base de données, sans état partagé applicatif.

---

## Alternatives envisagées

### Verrou optimiste avec champ `version`

Ajout d'un champ `version INTEGER` sur `Event`. Chaque mise à jour vérifie `WHERE version = :expectedVersion` et incrémente. En cas de conflit, la transaction échoue et le service relance jusqu'à N fois.

Raison de l'écart : sous forte concurrence (100+ requêtes simultanées pour le dernier ticket), le taux d'échec et de retry explose. Chaque retry rallonge la latence perçue et multiplie les lectures en base. La gestion applicative des conflits ajoute de la complexité sans garantie de terminaison bornée. Le verrou optimiste est adapté aux conflits rares — ce n'est pas le cas lors d'une ouverture de billetterie.

### File de réservation (queue-based allocation)

Les demandes d'inscription sont placées dans une file RabbitMQ. Un worker unique consomme séquentiellement et alloue les places sans concurrence.

Raison de l'écart : introduit un composant supplémentaire avec sa propre fiabilité à garantir (le worker). Le délai de traitement devient imprévisible sous charge. La complexité opérationnelle — monitoring de la file, gestion des messages en DLQ, scaling du worker — est disproportionnée pour le volume attendu de SupEvents (événements étudiants, capacités de quelques centaines de places). Cette architecture serait pertinente à l'échelle de Ticketmaster, pas ici.

### Redis DECR + Lua script

Utilisation de `DECR` atomique Redis sur un compteur `seats:{eventId}` pour pré-réserver la place, puis écriture différée en PostgreSQL.

Raison de l'écart : crée un risque de désynchronisation entre Redis (source de vérité temporaire) et PostgreSQL (source de vérité durable) en cas de crash entre les deux écritures. La réconciliation périodique requise ajoute de la complexité. Redis est déjà utilisé pour d'autres usages (sessions, idempotence) — lui confier la logique d'allocation critique augmente le couplage et la surface de défaillance.

---

## Conséquences

### Positives

- Correction garantie par le moteur de base de données, sans logique applicative complexe.
- Comportement prévisible et testable : les tests d'intégration avec transactions sont directs.
- Rollback automatique en cas d'erreur à n'importe quelle étape de l'allocation.
- Lisibilité du code : la logique tient en une transaction SQL claire.

### Négatives

- Le `SELECT FOR UPDATE` sérialise les inscriptions à un même événement : le débit maximal d'allocations est limité par les capacités de PostgreSQL sur cette table (estimé à ~500-1000 TPS sur le matériel cible, largement suffisant pour SupEvents).
- Risque de deadlock si d'autres transactions accèdent aux mêmes lignes dans un ordre différent — à mitiger par un ordre d'acquisition des verrous cohérent et un `lock_timeout` PostgreSQL configuré.
- La latence p99 lors des pics d'ouverture peut augmenter si des transactions longues retiennent le verrou — à surveiller via métriques.

### Dette acceptée

Le `lock_timeout` PostgreSQL n'est pas encore configuré dans l'environnement de développement. Sans ce paramètre, une transaction bloquée peut retenir le verrou indéfiniment. Cette dette devra être adressée avant la mise en production via la variable `SET lock_timeout = '2s'` en début de transaction, avec un code d'erreur `DB_LOCK_TIMEOUT` géré côté application (voir §7.1 — TicketModule).
