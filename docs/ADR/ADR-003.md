# ADR-003 — Les événements asynchrones RabbitMQ utilisent un backoff exponentiel avec dead letter queue dédiée

**Statut** : Accepté  
**Date** : 2025-05-13  
**Auteurs** : Sherazad

---

## Contexte

SupEvents s'appuie sur RabbitMQ pour découpler les modules via des événements asynchrones (`ticket.confirmed`, `payment.failed`, `event.cancelled`). Les consommateurs (NotificationService, AnalyticsService, WaitingListService) peuvent échouer à traiter un message pour des raisons transitoires (base de données surchargée, SendGrid indisponible, timeout réseau) ou permanentes (payload invalide, bug logiciel).

Sans politique de retry explicite, un message en échec est soit perdu (aucune durabilité), soit retraité indéfiniment (boucle infinie), soit bloquant (la file entière est paralysée en attendant le traitement du message problématique).

La contrainte principale est la suivante : les consommateurs doivent être idempotents (voir ADR-002) pour supporter les retrys sans effets de bord. La politique de retry doit distinguer les erreurs transitoires (à réessayer) des erreurs permanentes (à isoler).

---

## Décision

Nous adoptons un backoff exponentiel avec un maximum de 5 tentatives pour les événements critiques (`ticket.confirmed`, `payment.failed`) et 3 tentatives pour les événements secondaires (`event.cancelled`). Après épuisement des tentatives, le message est routé vers une dead letter queue (DLQ) dédiée par type d'événement (`tickets.events.dlq.v1`, `payments.events.dlq.v1`). Une alerte PagerDuty est déclenchée pour tout message en DLQ. Les DLQ sont rejouables manuellement après correction.

---

## Alternatives envisagées

### Retry immédiat à nombre fixe de tentatives

En cas d'échec, le message est remis immédiatement dans la file, jusqu'à N fois, puis dropé ou mis en DLQ.

Raison de l'écart : un retry immédiat après un échec dû à une indisponibilité de dépendance (ex : SendGrid en panne) va échouer à nouveau immédiatement, saturant le consommateur et la file. Sans intervalle croissant, les N tentatives s'enchaînent en quelques millisecondes et n'offrent aucune chance de récupération. Cette approche est correcte uniquement pour les erreurs non liées à des dépendances externes.

### Retry piloté par un scheduler externe (cron)

Les messages en échec sont persistés en base de données avec un statut `failed` et une date de prochain essai. Un job cron les relit et les retraite.

Raison de l'écart : introduit une dépendance forte entre la couche de messaging et la base de données, complexifiant le déploiement et la reprise sur incident. Le scheduler devient un point de défaillance supplémentaire. Cette architecture est adaptée aux jobs de traitement batch (réconciliation de paiements orphelins), pas aux événements temps-réel qui doivent déclencher des notifications dans les secondes suivant la confirmation d'un ticket.

### Délai linéaire fixe entre chaque retry

Intervalle constant entre les tentatives (ex : 5 secondes à chaque fois).

Raison de l'écart : meilleur que le retry immédiat, mais sous-optimal. Pour une panne prolongée (> 1 minute), un délai linéaire de 5s génère autant de tentatives inutiles qu'un retry rapide sur une fenêtre longue. Le backoff exponentiel réduit la pression sur la dépendance défaillante de façon adaptative, ce qui est le comportement recommandé dans les systèmes distribués (AWS, Google Cloud, Stripe utilisent tous ce pattern).

---

## Conséquences

### Positives

- Protection des dépendances externes (SendGrid, PostgreSQL) pendant les incidents : la charge de retry décroît automatiquement.
- Isolation des messages problématiques en DLQ : un message invalide ne bloque pas le traitement des messages suivants.
- Rejeu manuel possible après correction du bug ou rétablissement de la dépendance.
- Observabilité : la taille de la DLQ est un indicateur de santé directement monitorable.

### Négatives

- La latence maximale de traitement d'un événement augmente : avec 5 tentatives et un backoff exponentiel 1s/2s/4s/8s/16s, un message peut attendre jusqu'à 31 secondes avant d'atterrir en DLQ. Pour `ticket.confirmed`, cela signifie que l'email de confirmation peut être retardé de 30 secondes en cas de panne SendGrid courte.
- La configuration RabbitMQ (exchanges de retry, TTL des messages, binding DLQ) est plus complexe qu'une file simple et requiert une expertise opérationnelle spécifique.
- Le monitoring de la DLQ et les procédures de rejeu doivent être documentés et testés — sans quoi des messages s'y accumulent silencieusement.

### Dette acceptée

La procédure de rejeu manuel des messages en DLQ n'est pas encore documentée ni automatisée. À ce stade, le rejeu nécessite un accès direct à la console d'administration RabbitMQ ou un script CLI ad hoc. Une interface d'administration interne pour rejouer les DLQ par type d'événement est identifiée comme amélioration prioritaire post-MVP, mais reportée au semestre suivant.
