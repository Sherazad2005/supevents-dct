# ADR-002 — L'idempotence des opérations d'écriture est garantie par déduplication serveur via Redis

**Statut** : Accepté  
**Date** : 2025-05-13  
**Auteurs** : Sherazad

---

## Contexte

Plusieurs opérations critiques de SupEvents sont susceptibles d'être rejouées : les appels clients en cas de timeout réseau (`POST /tickets`, `POST /payments/initiate`), les webhooks Stripe qui livrent les événements en *at-least-once*, et les messages RabbitMQ qui peuvent être rédélivrés après un crash consommateur.

Sans mécanisme d'idempotence, un double appel à `POST /payments/initiate` pourrait créer deux `PaymentIntent` Stripe distincts et débiter l'utilisateur deux fois. Un webhook `payment_intent.succeeded` rejoué confirmerait le ticket deux fois et enverrait deux emails de confirmation.

La question est : qui porte la responsabilité de l'idempotence — le client (via une clé d'idempotence fournie) ou le serveur (via une déduplication automatique) ?

---

## Décision

Nous utilisons une déduplication serveur systématique via Redis. Pour les appels API, la clé d'idempotence est construite côté serveur à partir d'un identifiant métier stable (`ticket_id`, `event.id` Stripe). Pour les webhooks, la clé est l'`event.id` fourni par Stripe. Les clés sont stockées dans Redis avec un TTL adapté (24h pour les paiements, 48h pour les webhooks).

---

## Alternatives envisagées

### Clé d'idempotence client (pattern Stripe)

Le client génère un UUID aléatoire et l'envoie dans un header `Idempotency-Key`. Le serveur stocke la réponse associée à cette clé et la rejoue en cas de renvoi.

Raison de l'écart : transfère la responsabilité au client, qui doit stocker et réutiliser sa clé entre les tentatives. En contexte mobile ou SPA avec reconnexion réseau, cette clé est souvent perdue entre deux tentatives, rendant la protection inefficace. De plus, un client malveillant pourrait générer une nouvelle clé pour forcer un deuxième traitement. La déduplication serveur est plus robuste dans notre contexte.

### Contrainte d'unicité base de données uniquement

Utilisation d'une contrainte `UNIQUE(ticket_id)` sur la table `Payment` pour rejeter les doublons au niveau SQL, sans Redis.

Raison de l'écart : cette contrainte est nécessaire (et présente dans notre schéma) mais insuffisante comme seul mécanisme. Elle ne couvre pas les cas où la requête Stripe a été émise mais l'insertion en base n'a pas eu lieu (crash entre les deux). Elle ne couvre pas non plus l'idempotence des webhooks, qui portent un `event.id` distinct même pour le même `PaymentIntent`. Redis offre une couche de protection complémentaire et plus flexible.

### Idempotence stateless via signature du payload

Construction d'une clé de déduplication à partir du hash du payload de la requête (HMAC ou SHA-256). Stockage en base de données.

Raison de l'écart : le hash du payload change si le client envoie des données légèrement différentes (timestamp, champ optionnel), rendant la déduplication fragile. Le stockage en base de données est plus lent que Redis pour ce type de lookup à haute fréquence. La construction d'une clé métier stable est préférable à un hash de payload.

---

## Conséquences

### Positives

- Protection transparente pour le client : aucune responsabilité supplémentaire côté front.
- Couverture complète : API, webhooks et messages RabbitMQ utilisent le même mécanisme.
- Performances : le lookup Redis est sub-milliseconde, sans impact perceptible sur la latence.
- Simplicité opérationnelle : Redis est déjà dans l'infrastructure pour les sessions et le rate limiting.

### Négatives

- Redis devient un composant critique sur le chemin des paiements. Sa disponibilité conditionne l'idempotence — une panne Redis sans fallback pourrait laisser passer des doublons.
- Le TTL doit être dimensionné avec soin : trop court (< délai de retry Stripe), des replays légitimes ne sont pas dédupliqués ; trop long, la mémoire Redis est gaspillée.
- La clé d'idempotence construite côté serveur suppose que l'identifiant métier (`ticket_id`) est disponible dès l'appel initial — ce qui contraint l'ordre des opérations.

### Dette acceptée

Le comportement en cas d'indisponibilité Redis sur le chemin d'idempotence n'est pas encore spécifié : faut-il rejeter la requête (`fail-closed`) ou la laisser passer en acceptant le risque de doublon (`fail-open`) ? La décision `fail-closed` est plus sûre pour les paiements mais dégrade la disponibilité. Cette arbitrage doit être documenté et testé avant la mise en production.
